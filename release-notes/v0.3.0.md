# Release v0.3.0

Release highlights

- Remediation of all CODE_REVIEW.md findings across 10 focused change units. ✅
- Security hardening: API key values are no longer logged at any level; `x-api-key` header redacted; throttler now keys by Bearer token for authenticated routes. 🔒
- Correctness fixes in the IMAGE assessment path: broken Buffer support fixed, image ordering in the system prompt corrected, JSON parser made robust against trailing braces in LLM output. 🐛
- Performance improvements: template disk reads now cached, MIME-type lookups use `Set`, structured pino logging avoids unconditional string concatenation. ⚡
- Image pipeline refactored: `ImagePrompt` no longer reads files from disk — `PromptFactory` converts Buffers to data URIs upstream. Removed the `images` array from `CreateAssessorDto` (clients must supply image data directly in `reference`, `studentResponse`, `template` fields). 🏗️

## Details

### Security 🔒

- **API key logging eliminated** — `ApiKeyService` no longer logs key values at WARN or DEBUG level; only opaque failure messages are emitted. Both branches use the same message to prevent a format oracle.
- **`x-api-key` header redacted** — `LogRedactor` now masks the `x-api-key` header in request logs (alongside the existing `authorization`/`cookie` redaction).
- **Per-API-key rate limiting** — New `ApiKeyThrottlerGuard` keys throttling by Bearer token for authenticated requests, falling back to IP for unauthenticated ones. Each API key now has its own independent rate-limit counter.
- **`ConfigService` for environment access** — `ZodValidationPipe`, `HttpExceptionFilter`, and `app.module.ts` now read `NODE_ENV` (and `LOG_FILE`) through `ConfigService` instead of `process.env`. The `LOG_FILE` variable has been added to the Zod environment schema.
- **Unused dependencies removed** — `@modelcontextprotocol/sdk` and `@openai/codex-sdk` removed from production dependencies, reducing the supply-chain attack surface.

### Bug fixes 🐛

- **Broken Buffer IMAGE path fixed** — `PromptFactory` now correctly converts Buffer inputs to data URIs using `detectBufferMime` + `Buffer.toString('base64')` instead of default UTF-8 `.toString()`. The documented Buffer support now works, returning proper 400s on invalid input rather than HTTP 500s.
- **Image system prompt ordering corrected** — `image.system.prompt.md` Step 1 now correctly describes `Template: {description of the second image}` and `Student Submission: {description of the third image}`, matching the actual data-URI ordering (`[reference, template, student]`).
- **JSON parser made robust** — Replaced the naive first-brace-to-last-brace trim with a string-aware balanced-brace scanner that correctly handles braces inside string values and trailing prose containing `}` characters.
- **`ImageValidationPipe` now rejects non-data-URI strings** — Instead of silently passing them through (which triggered a raw `Error` → HTTP 500 downstream), the pipe now throws a clear `BadRequestException`.
- **ZodError logged correctly** — `gemini.service.ts` now stringifies `error.issues` instead of passing an array as the `stack` argument.
- **Empty leading text part removed** — `buildContents` no longer prepends an empty string to multimodal image contents.

### Performance ⚡

- **Template caching** — `PromptFactory` now caches template contents in a `Map` keyed by filename, eliminating redundant disk reads on every request.
- **`Set`-based lookups** — API keys stored in a `Set` for O(1) membership checks (`api-key.service.ts`); allowed MIME types pre-computed as a `Set` in `ImageValidationPipe`.
- **Duplicate computation eliminated** — `GeminiService.generateAndParseResponse` now receives pre-built `modelParameters` and `contents`, avoiding a second call to `buildModelParams`/`buildContents` on the hot path.
- **Structured logging** — Debug-level log calls on the hot path pass raw objects to pino (`{ contents }`, `{ responseText }`, `{ parsedJson }`) instead of pre-stringifying with `JSON.stringify(…, null, 2)`.

### Image pipeline refactoring 🏗️

- **`ImagePrompt` simplified** — Removed file-based image reading (`readImageFile`, `buildImagesFromFiles`). The class now only handles data URI input (including Buffers converted to data URIs upstream by `PromptFactory`).
- **`PromptFactory` Buffer → data URI conversion** — `instantiatePrompt` for `TaskType.IMAGE` now detects MIME types via `detectBufferMime` and constructs `data:{mimeType};base64,{data}` URIs from Buffer inputs.
- **`images` field removed from DTO** — The `images: { path: string, mimeType: string }[]` field has been removed from `CreateAssessorDto`. Image data must now be supplied via the `reference`, `studentResponse`, and `template` Buffer/string fields.

### Linting and compliance 📐

- **British-English wordlist expanded** — `check-british-english.sh` now catches `behavior`, `prioritize`, `sanitize`, `catalog`, `grey`/`gray`.
- **Linter override authorisation documented** — All suppression comments now include inline justification referencing `docs/development/linter-overrides.md`. A new doc explains the authorisation process.
- **ESLint config updated** — `unicorn/prefer-uint8array-base64` override expanded to include `prompt.factory.ts` and `prompt.factory.spec.ts` (the Node.js runtime does not yet ship `Uint8Array.prototype.toBase64()`).

### Error detection robustness 🔄

- **`LLMService` error detection broadened** — `extractErrorStatusCode` now reads `statusCode`, `code`, `error.status`, `error.code`, `response.status`, and string values that parse to a number (e.g. `'429'`). `isResourceExhaustedError` and `isRateLimitError` recognise gRPC-style string statuses (`'RESOURCE_EXHAUSTED'`, `'RATE_LIMIT_EXCEEDED'`).
- **New unit tests** — `llm.service.interface.spec.ts` (272 lines) provides comprehensive coverage of the error detection helpers.

## Known limitations and notes

- **`ApiKeyThrottlerGuard`** is a custom `ThrottlerGuard` subclass; it relies on `@nestjs/throttler` v6 and may need review when upgrading the throttler package.
- **MIME detection** — `mime-detect` requires the OS `file` binary (typically available on Linux, macOS). Tests using `detectBufferMime` will fail in environments without it. The binary was already a documented dependency.
- **`unicorn/prefer-uint8array-base64` override** — Retained because Node.js 24 does not yet ship the `Uint8Array.prototype.toBase64()`/`fromBase64()` methods at runtime, despite TypeScript's lib declarations. This is a known Node.js version gap.

## Upgrade notes

- **Breaking change**: the `images` array (`{ path, mimeType }[]`) has been removed from `CreateAssessorDto`. Clients using the IMAGE task type must now supply image data directly in the `reference`, `studentResponse`, and `template` fields (as Buffers or data-URI strings) instead of referencing file paths. The old file-based `ImagePrompt.readImageFile` path has been removed.
- **Rate-limiting behaviour**: authenticated requests are now throttled per API key (Bearer token) rather than per IP address. If you monitor rate-limit counters, note that each key now has its own independent counter.
- **Logging changes**: API key values are no longer logged at any level. If you relied on DEBUG-level log analysis for troubleshooting authentication failures, you will now see only opaque failure messages (`Authentication failed: invalid API key presented`).
- **Import changes**: no import-level changes in this release.
- **Dependency changes**: `@modelcontextprotocol/sdk` and `@openai/codex-sdk` have been removed from production dependencies. Run `npm ci` after pulling to ensure a clean tree.
- **Linter notes**: if you extend the project's ESLint config, note that the `unicorn/prefer-uint8array-base64` override now covers two additional files (`prompt.factory.ts`, `prompt.factory.spec.ts`). The rule remains disabled for files that use `Buffer.prototype.toString('base64')` due to the Node.js runtime gap described above.
