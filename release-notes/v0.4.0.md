# Release v0.4.0

Release highlights

- Centralised LLM error handling library introduced (`src/common/errors/`). 🏗️
- 5xx upstream errors are now retried (behaviour change — see upgrade notes). 🔄
- New `LlmError`-based error hierarchy with nine subclasses. 🎯
- `ResourceExhaustedError` migrated into the shared hierarchy. 📦
- Comprehensive error-handling guide published at `docs/llm/error-handling.md`. 📝

## Details

### Error handling 🏗️

- **Centralised library created** — `src/common/errors/` contains the abstract
  `LlmError` base class (extending `HttpException`) and nine concrete subclasses,
  all re-exported via `src/common/errors/index.ts`. Error classes are plain
  TypeScript and imported statically — no NestJS module wrapper is needed.
- **`ResourceExhaustedError` migrated** — formerly a plain `Error` subclass at
  `src/llm/resource-exhausted.error.ts`, it now extends `LlmError` with a
  hardcoded HTTP 503 and `retryable = false`. The old file has been removed.
- **Abstract `mapError()` contract** — each `LLMService` subclass implements
  `mapError(error: unknown): LlmError | undefined` and declares
  `protected readonly providerName: string`.
- **`GeminiService.mapError()` implemented** — maps all nine error categories
  from Gemini SDK error shapes, following the agreed classification priority
  order. Any unrecognised 4xx maps to `InvalidRequestError` (HTTP 400), not
  `undefined`, preserving 4xx semantics for API consumers.
- **Retry loop refactored** — the base `LLMService.send()` now checks
  `ZodError` first (re-throws without retry), then calls `mapError()` and
  branches on the returned `LlmError.retryable` flag. Old private detection
  helpers (`isRateLimitError`, `isResourceExhaustedError`,
  `extractErrorStatusCode`, etc.) have been removed.
- **`HttpExceptionFilter` simplified** — the explicit
  `instanceof ResourceExhaustedError` branch has been removed; the migrated
  error is handled by the generic `HttpException` branch, which applies the
  production 5xx sanitisation gate consistently.

### Documentation 📝

- New dedicated guide `docs/llm/error-handling.md` covering the full error
  hierarchy, classification priority rules, provider integration steps, error
  message policy, `originalError` narrowing, and a worked example.
- `docs/modules/llm.md` updated to remove the stale `ResourceExhaustedError`
  snippet and point to the new centralised guide.
- `AGENTS.md` now references `docs/llm/error-handling.md` for adding providers
  or error types.

### 5xx-retryable behaviour change 🔄

- **Before:** Any non-429 upstream error (including HTTP 500, 502, 503) caused
  `LLMService.send()` to fail fast after a single attempt and throw a generic
  `Error` with HTTP 500.
- **After:** `ProviderServerError` (HTTP 502, `retryable = true`) and
  `NetworkError` (HTTP 502, `retryable = true`) are retried up to
  `LLM_MAX_RETRIES` attempts with exponential backoff. The final thrown error
  carries HTTP 502 (not 500).

### Testing ✅

- New unit tests for each error class (HTTP status, `retryable` flag,
  inheritance chain).
- Comprehensive `GeminiService.mapError()` tests covering all nine error
  categories, priority conflicts, non-object/`null` inputs, and the
  unrecognised-4xx → `InvalidRequestError` default.
- Base-class `LLMService` retry-loop tests rewritten to verify retry/throw
  behaviour against `LlmError.retryable` flags.
- `HttpExceptionFilter` tests updated for the simplified error handling path.
- All 329 tests passing.

## Known limitations and notes

- **`waitBeforeRetry()` log message** — The warn-level message reads
  `"Rate limit encountered on attempt ..."` even when the retryable error is a
  `ProviderServerError` or `NetworkError`. This pre-existing wording was kept
  verbatim per product decision #6 (see SPEC.md); it may be generalised in a
  future release.
- **429 message exposure** — `RateLimitError` (429) messages are exposed
  unsanitised in production, per product decision #5 (see SPEC.md). The
  diagnostic value outweighs the low-sensitivity leak surface of
  provider-internal identifiers in the response body.
- **`ZodError` HTTP behaviour unchanged** — `ZodError` (validation failure)
  continues to bypass `mapError()` and flows through the generic `Error`
  fallback in `HttpExceptionFilter`, producing HTTP 500 (sanitised to
  `"Internal server error"` in production). Changing `ZodError` to produce a
  400 is deferred.

## Upgrade notes

- **Breaking: 5xx upstream errors now return 502, not 500.** Clients that
  previously relied on HTTP 500 for upstream 5xx responses (e.g. monitoring
  alerts, retry logic in API consumers) will now receive HTTP 502 with
  `ProviderServerError` or `NetworkError` instead. The error type is now an
  `LlmError` subclass rather than a generic `Error`.
- **Breaking: `ResourceExhaustedError` is no longer a plain `Error`.** It now
  extends `HttpException` (via `LlmError`). Any code using
  `instanceof ResourceExhaustedError` will still work (the class name is
  unchanged), but the inheritance chain is different. The import path has
  changed to `src/common/errors/index.js`.
- **Breaking: old private detection helpers removed.** If any external code
  referenced `isRateLimitError()`, `isResourceExhaustedError()`,
  `extractErrorStatusCode()`, `matchesStringStatus()`, `handleSendError()`,
  `throwTerminalSendError()`, or `buildUnexpectedErrorMessage()` from
  `LLMService`, those references will no longer compile.
- **Import changes:** `ResourceExhaustedError` must now be imported from
  `src/common/errors/index.js` instead of `src/llm/resource-exhausted.error.js`.
  The old file has been deleted.
